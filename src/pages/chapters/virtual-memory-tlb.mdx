# 가상 메모리와 TLB

## 모든 프로그램이 같은 주소 0x400000을 쓰는데 안 부딪힌다?

서로 다른 두 프로그램을 디버거로 까 보면 둘 다 비슷한 주소(예: `0x400000`)에 코드를 올려
둡니다. 그런데 둘은 서로의 메모리를 망가뜨리지 않습니다. 한 프로그램이 잘못된 포인터를
건드리면 *그 프로그램만* 죽고 옆은 멀쩡합니다. 어떻게 같은 주소가 충돌하지 않을까요?

답은 프로그램이 보는 주소가 **가상주소(virtual address)** 이고, 실제 DRAM의 **물리주소
(physical address)** 와 다르기 때문입니다. 각 프로세스는 자기만의 가상주소 공간을 갖고,
하드웨어+OS가 매 접근마다 가상→물리로 **변환(translation)** 합니다. 이 챕터는 그 변환이
어떻게 일어나고, 왜 그게 느릴 수 있으며, **TLB** 가 어떻게 그걸 구해 주는지를 다룹니다.

## 페이지 — 변환의 단위

가상주소를 byte 하나하나 매핑하면 매핑 표가 메모리만큼 커집니다. 그래서 주소 공간을 고정
크기 **페이지(page)** 로 나누고 페이지 단위로 매핑합니다. x86-64의 기본 페이지는 **4 KB**
입니다. 가상주소의 하위 12 bit는 *페이지 안에서의 offset*(2¹² = 4096)이고, 변환되는 건
나머지 상위 비트(페이지 번호)뿐입니다 — offset은 변환을 그대로 통과합니다.

페이지 번호 → 물리 프레임 번호를 적어 둔 표가 **페이지 테이블(page table)** 입니다. 그런데
48-bit 가상주소에서 페이지 번호는 36 bit — 단순 배열이면 엔트리 2³⁶개로 한 프로세스당
수백 GB짜리 표가 됩니다. 말이 안 됩니다.

## 다단계 페이지 테이블 — 희소한 주소 공간을 위한 트리

실제 프로그램은 그 거대한 주소 공간 중 *극히 일부*만 씁니다(코드·힙·스택 몇 군데). 그래서
페이지 테이블을 **트리(다단계, multi-level)** 로 만들어, 쓰지 않는 가지는 아예 만들지
않습니다. x86-64는 페이지 번호 36 bit를 **9·9·9·9** 로 쪼개 4단계 트리로 만듭니다(각
9 bit = 512개 엔트리 = 한 페이지에 딱 맞음):

$$
\underbrace{9}_{\text{PML4}}\ \big|\ \underbrace{9}_{\text{PDPT}}\ \big|\ \underbrace{9}_{\text{PD}}\ \big|\ \underbrace{9}_{\text{PT}}\ \big|\ \underbrace{12}_{\text{offset}}
$$

각 단계의 테이블 엔트리는 *다음 단계 테이블의 물리주소*를 담고, 마지막 PT의 엔트리(PTE)가
실제 물리 프레임을 가리킵니다. 트리의 뿌리(PML4)의 물리주소는 **`CR3`** 레지스터에 들어
있고, 컨텍스트 전환 때 이 레지스터를 갈아 끼우면 곧 *다른 주소 공간으로* 바뀝니다.

이 트리를 따라 내려가 물리주소를 알아내는 과정이 **page table walk** 입니다. 핵심 비용이
여기서 드러납니다: 캐시에 없으면 변환 **한 번**이 메모리 **4번** 접근입니다(5-level이면 5번).
정작 원하는 데이터를 읽기도 전에 말이죠.

> **5-level paging.** working set이 더 큰 서버를 위해 Intel은 **5-level paging**(LA57)을
> 도입했습니다 — PML5가 한 단계 더 붙어 가상주소가 **48→57 bit**(주소 공간 256 TiB →
> 128 PiB), 물리주소는 최대 52 bit가 됩니다. `CR4`의 LA57 비트로 켜며, Ice Lake 이후
> 서버에서 지원합니다. 켜지면 walk가 5단계가 됩니다.

## page walk는 얼마나 비싼가

walk가 매번 메모리 4번이면 가상 메모리는 프로그램을 4배 느리게 만들 것입니다. 두 장치가
이를 구합니다.

첫째, walk가 읽는 페이지 테이블 엔트리들 자체가 **일반 데이터 캐시(L1/L2/L3)** 에 캐시될
수 있습니다. 그리고 코어에는 walk 중간 단계를 전용으로 캐싱하는 **page-walk cache(PWC)**
가 있어, 상위 단계(PML4/PDPT)는 거의 항상 적중합니다. 그래서 평균 walk 비용은 "메모리 4회"가
아니라 그보다 훨씬 적습니다(연구에 따라 평균 1.5회 미만 보고). 그래도 최악의 random 접근에선
walk가 수백 cycle까지 갈 수 있고, TLB miss 처리에 CPU 시간의 큰 비중을 쓰는 워크로드도 있습니다.

둘째, 그리고 더 중요한 것 — 방금 한 변환의 *결과* 를 캐싱하는 **TLB** 입니다.

## TLB — 변환의 캐시

**TLB(Translation Lookaside Buffer)** 는 "가상 페이지 번호 → 물리 프레임 번호" 매핑을 담는
작고 빠른 캐시입니다. 데이터 캐시가 *데이터* 를 캐싱하듯, TLB는 *변환* 을 캐싱합니다. 코어가
주소에 접근하면 먼저 TLB를 봅니다 — 있으면(**TLB hit**) walk 없이 즉시 물리주소를 얻고,
없으면(**TLB miss**) walk를 돌린 뒤 그 결과를 TLB에 채웁니다.

TLB도 캐시처럼 계층적입니다. 작고 빠른 **L1 (D)TLB**, 더 크고 느린 **L2 (S)TLB**. 대표값으로
Intel Skylake는 L1 DTLB가 4 KB 페이지 기준 64 엔트리, L2 STLB는 명령·데이터를 공유하는
약 1536 엔트리입니다(세대·구현마다 다름).

TLB hit는 사실상 공짜(주소 계산에 묻힘)이고 miss는 walk만큼 비쌉니다. 그래서 메모리 접근
패턴이 좋아 TLB가 잘 맞으면 가상 메모리는 거의 공짜처럼 느껴지고, working set이 TLB 용량을
넘겨 **TLB thrashing** 이 나면 walk 비용이 그대로 드러납니다.

## page fault — 매핑이 아예 없을 때

TLB miss는 "변환이 TLB에 *없을 뿐*, 페이지 테이블엔 있다"는 상황입니다. 그런데 walk를
돌렸는데 페이지 테이블 엔트리가 *유효하지 않으면*(매핑 자체가 없거나 권한 위반) 하드웨어는
**page fault** 예외를 던지고 OS로 넘깁니다. 이건 하드웨어가 혼자 못 푸는, OS가 개입하는
사건입니다. OS는 상황을 보고:

- **demand paging**: 아직 메모리에 안 올린 페이지면 디스크/스왑에서 읽어 와 매핑하고 재시도
  (정상적인 lazy 할당). 이건 흔하고 정상입니다.
- **copy-on-write**: `fork` 후 공유 페이지에 쓰려는 순간 fault → 복사본을 만들어 매핑.
- **잘못된 접근**: 매핑할 근거가 없으면(널 포인터 등) 프로세스를 죽입니다 — 이것이
  segmentation fault입니다.

가상 메모리가 디스크를 물리 메모리의 연장으로 쓰고(스왑), 프로세스를 서로 격리하고, lazy
할당을 하는 모든 기능이 이 page fault 핸들러 위에서 돕니다.

## huge / large page — TLB reach를 키우기

TLB 엔트리 하나는 페이지 *하나* 의 변환만 담습니다. 그러니 TLB가 한 번에 덮는 메모리
(**TLB reach**)는 "엔트리 수 × 페이지 크기"로 묶입니다. 큰 데이터를 4 KB 페이지로 다루면
페이지 수가 엄청나 TLB가 금세 넘칩니다.

해법은 **페이지를 키우는 것** 입니다. x86-64는 walk 중간 단계에서 트리를 *일찍 끝내* 큰
페이지를 만들 수 있습니다: PD 엔트리에서 멈추면 **2 MB**(다단계의 마지막 9 + 12 bit가
offset이 됨), PDPT 엔트리에서 멈추면 **1 GB** 페이지입니다. 한 엔트리가 4 KB의 512배,
512²배를 덮으니 TLB reach가 폭발적으로 커집니다.

<HugePageCoverage client:visible />

DB·과학계산·가상화처럼 working set이 수십 GB인 워크로드는 huge page로 TLB miss를 극적으로
줄입니다. 단점도 있습니다: 큰 페이지는 연속된 물리 메모리가 필요해 단편화에 약하고,
2 MB를 통째로 올리니 내부 단편화·메모리 낭비가 생길 수 있습니다. Linux의 **Transparent
Huge Pages(THP)** 는 OS가 알아서 4 KB를 2 MB로 합쳐 주는 절충안입니다.

## VIPT 캐시 — 변환과 캐시 조회를 겹치기 (한 줄)

L1 캐시가 데이터를 찾으려면 물리주소가 필요한데, 그 물리주소를 얻으려면 TLB를 봐야 합니다 —
직렬로 하면 느립니다. 그래서 L1은 보통 **VIPT(Virtually Indexed, Physically Tagged)** 입니다:
*가상주소의 하위 비트(offset, 변환을 안 거침)로 캐시 set을 고르는 일* 과 *TLB 조회* 를
**병렬로** 하고, TLB가 준 물리 tag로 set 안에서 비교만 합니다. 변환과 캐시 인덱싱을 겹쳐
지연을 숨기는 트릭입니다(단, set index가 offset 비트 안에 들어가도록 캐시 크기에 제약이 생깁니다).

## 컨텍스트 전환과 TLB — flush vs ASID

TLB 엔트리는 *특정 프로세스의* 가상→물리 매핑입니다. 프로세스를 바꾸면(컨텍스트 전환,
`CR3` 교체) 그 매핑은 무효 — 옛날엔 전환 때마다 TLB를 통째로 **flush** 했습니다. 그러면
새 프로세스는 TLB가 텅 빈 채로 시작해 한동안 walk 폭풍을 맞습니다(컨텍스트 전환 비용의
숨은 항목). 전체 무효화 비용도 수천 cycle에 이를 수 있습니다.

이를 줄이려고 각 TLB 엔트리에 **주소 공간 식별자**(ARM: **ASID**, x86: **PCID**)를 태그로
붙입니다. 그러면 여러 프로세스의 변환이 TLB에 *공존* 할 수 있고, 전환 때 flush 없이 식별자만
바꾸면 됩니다 — 돌아온 프로세스는 자기 옛 엔트리를 그대로 재사용합니다. OS 스케줄링과 전환
비용은 [`./os-scheduling-context-switch`](./os-scheduling-context-switch)에서 더 다룹니다.

> **주의(보안).** Meltdown 완화책 KPTI(kernel page-table isolation)는 user/kernel 페이지
> 테이블을 분리해 전환 때 TLB 압박을 늘렸습니다 — PCID가 그 비용을 일부 덜어 줍니다. 정확한
> 동작·비용은 세대·OS·완화 설정에 따라 다릅니다.

## 종합

- 프로그램은 **가상주소** 를 보고, 하드웨어가 페이지 단위로 **물리주소** 로 변환한다.
- 변환표는 희소성을 위해 **다단계 트리**(x86-64 4-level, 또는 5-level)이고, walk는 캐시되지
  않으면 메모리 여러 번 접근이라 비싸다.
- **TLB**(L1/L2)가 변환 결과를 캐싱해 hit를 공짜로 만든다. miss면 walk(PWC가 도움).
- 매핑이 없으면 **page fault** → OS가 demand paging·CoW·격리(segfault)를 처리한다.
- **huge page**(2 MB/1 GB)는 TLB reach를 키워 큰 working set의 miss를 줄인다.
- **VIPT** 로 변환·캐시 인덱싱을 겹치고, **ASID/PCID** 로 컨텍스트 전환 시 TLB flush를 피한다.

## 더 나아가기

- **캐시와의 관계:** TLB는 *주소 변환* 캐시, L1/L2/L3는 *데이터* 캐시 — 큰 working set은 둘
  다 thrash한다 → [`./cpu-memory-hierarchy`](./cpu-memory-hierarchy).
- **컨텍스트 전환 비용:** TLB flush/ASID는 전환 비용의 한 항목 →
  [`./os-scheduling-context-switch`](./os-scheduling-context-switch).
- **코어 간 TLB:** 한 코어가 매핑을 바꾸면 다른 코어의 TLB도 무효화해야 한다(**TLB
  shootdown**, IPI로) — 멀티코어에서 또 다른 비용.
- **5-level·물리주소 폭:** 서버 메모리 증가에 따라 주소 폭이 계속 늘고 있다(LA57, 52-bit PA).
