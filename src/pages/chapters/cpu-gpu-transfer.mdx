# CPU↔GPU 데이터 전송 — PCIe·DMA·Resizable BAR

[명령 큐](./command-queues)와 [CPU와 GPU의 대화](./gpu-cpu-conversation)에서 CPU가 GPU에 *명령*을
보내는 길을 봤습니다. 이 챕터는 *데이터* — 정점·텍스처·버퍼 — 가 시스템 RAM과 VRAM 사이를
건너는 길, 즉 **PCIe 버스**와 그 위의 전송 메커니즘을 봅니다. 이 구간은 의외로 자주 병목이 됩니다.

## PCIe — 직렬 점대점 버스

PCIe는 **직렬 점대점 링크**입니다. 레인(lane)을 묶어 폭을 키우며(GPU는 보통 **x16**), 각 레인은
송·수신 차동 쌍을 따로 가져 **전이중**(양방향 동시 풀 대역폭)입니다. 세대마다 레인 속도가 대략
2배씩 오릅니다.

| 세대 | 레인당 | x16 방향당(유효) |
|---|---|---|
| PCIe 3.0 | 8 GT/s (~1 GB/s) | **≈ 15.75 GB/s** |
| PCIe 4.0 | 16 GT/s (~2 GB/s) | **≈ 31.5 GB/s** |
| PCIe 5.0 | 32 GT/s (~4 GB/s) | **≈ 63 GB/s** |

GT/s는 인코딩을 포함한 원시 전이율이고, GB/s는 인코딩(3.0+는 128b/130b, 약 1.5% 오버헤드)을 뺀
페이로드입니다. 실측은 TLP 헤더·플로 제어 때문에 이론치의 80~90%대입니다. 멘탈 모델은 간단합니다:
**세대마다 대략 2배.**

## DMA와 pinned 메모리 — 왜 page-lock이 빠른가

대용량 전송은 CPU가 한 바이트씩 옮기지 않습니다. GPU의 **copy engine(DMA 엔진)**이 PCIe 너머로
블록을 **비동기로** 옮기고, 그동안 CPU는 자유롭습니다. 그런데 DMA 엔진은 **고정된 물리 주소**를
타깃해야 합니다. OS가 언제든 스왑·이동할 수 있는 일반 **pageable** 메모리는 직접 DMA 대상이 될 수
없습니다.

그래서 pageable 메모리를 전송하면, 드라이버가 **임시 pinned(page-locked) 버퍼를 만들어 거기로 한 번
복사한 뒤** DMA합니다. 이 추가 복사 때문에 pageable 전송은 느립니다. 처음부터 **pinned**로 할당하면
(`cudaMallocHost` 등) 이 단계가 사라져 PCIe 대역폭 상한에 근접합니다. 다만 pinned는 OS가 회수 못
하는 물리 RAM을 묶으므로 **과다 할당은 시스템을 느리게** 합니다 — 스테이징 영역으로 아껴 씁니다.

## BAR와 Resizable BAR — CPU가 VRAM을 보는 창

CPU가 VRAM에 직접 쓰려면 VRAM이 CPU의 주소 공간에 보여야 합니다. 그 창이 **BAR(Base Address
Register)**입니다. 전통적으로 BAR는 **256 MB**로 제한돼, VRAM이 수 GB라도 CPU는 한 번에 256 MB만
직접 봤습니다. **Resizable BAR**(AMD는 **Smart Access Memory**로 브랜딩)는 이 창을 **VRAM 전체**로
키웁니다.

게임 FPS 이득은 작고 가변적(평균 한 자릿수 %)이지만, 그래픽스 프로그래밍에서의 진짜 가치는 따로
있습니다 — CPU가 VRAM에 **직접 쓸 수 있게 되어 스테이징 복사를 없애는 것**입니다. 아래에서 ReBAR를
껐다 켜며 CPU-visible 창이 어떻게 커지는지, 어떤 힙이 열리는지 보세요.

## 메모리 타입과 힙 — 무엇을 어디에 둘까

API는 메모리를 용도별 **힙/타입**으로 나눕니다.

**D3D12.** `DEFAULT`(VRAM, GPU 전용), `UPLOAD`(시스템 RAM, CPU 쓰기·GPU 읽기 — 스테이징), `READBACK`
(GPU→CPU 읽기), 그리고 신규 `GPU_UPLOAD`(**VRAM이면서 CPU가 직접 쓰기** — ReBAR 필수, 스테이징
복사 제거).

**Vulkan.** `DEVICE_LOCAL`(VRAM), `HOST_VISIBLE`(CPU 매핑 가능), 그리고 특수 조합
`DEVICE_LOCAL | HOST_VISIBLE`(= BAR 메모리, ReBAR 없으면 보통 **~256 MB**로 제한). 매핑 잦으면
**persistent mapping**(매 프레임 Map/Unmap 금지)이 정석입니다.

전형적 패턴: **정적 자원**(메시·텍스처)은 `UPLOAD`/`HOST_VISIBLE` 스테이징 버퍼에 CPU가 쓰고
**한 번** `DEVICE_LOCAL`/`DEFAULT`로 복사한 뒤, 이후엔 GPU 로컬 풀밴드로만 접근합니다.

> **주의.** CPU가 쓰는 VRAM(`GPU_UPLOAD`·BAR 메모리)은 보통 **write-combined(uncached)**라
> **CPU에서 읽으면 매우 느립니다.** 쓰기 전용으로만 쓰고, readback은 `READBACK`/`HOST_CACHED`를 쓰세요.

## 통합 메모리 — UVM vs APU

**CUDA Unified Memory**(`cudaMallocManaged`)는 CPU·GPU가 **하나의 포인터**로 접근하게 합니다. 커널이
미상주 페이지를 건드리면 **page fault**가 나고 그 페이지를 GPU로 **마이그레이션**합니다(Pascal이 최초로
하드웨어 지원). 편하지만 폴트·마이그레이션 비용이 있어, 핫패스에선 `cudaMemPrefetchAsync`로 미리
옮깁니다.

이건 **이산(NUMA) GPU에서 소프트웨어/하드웨어가 자동 복사**해 통합처럼 보이게 하는 것입니다. 반면
**APU/내장 GPU**는 CPU와 **물리적으로 같은 RAM**을 쓰는 **통합(UMA)** 구조라 PCIe 복사 자체가
없습니다(진짜 zero-copy). 콘솔이 업로드 비용에서 자유로운 이유입니다.

## 실전 패턴

- **스테이징 업로드**: 정적 자원은 한 번 올리고 GPU에 상주. 매 프레임 큰 데이터를 host→device 하지 말 것.
- **transfer queue 오버랩**: 전용 copy 큐(비-디폴트 스트림)에서 전송을 돌려 그래픽스/컴퓨트와 병행.
  CUDA에선 **pinned + non-default stream**일 때만 copy+compute가 겹칩니다.
- **GPUDirect / RDMA**: 스토리지·NIC가 host RAM을 안 거치고 GPU 메모리에 직접 DMA — CPU I/O 병목 제거.

## 정리

- PCIe는 전이중 직렬 버스, 세대마다 ~2배(gen3/4/5 x16 ≈ 15.75/31.5/63 GB/s 방향당).
- DMA는 **pinned** 메모리를 요구 — pageable은 임시 pinned로 한 번 더 복사돼 느리다.
- **BAR**는 CPU가 VRAM을 보는 창(전통적 256 MB). **Resizable BAR**가 전체로 키워 `GPU_UPLOAD` 등
  **CPU 직접 쓰기 VRAM**을 열고 스테이징을 없앤다(단 CPU read 금물).
- 힙을 용도에 맞게(DEFAULT/UPLOAD/READBACK/GPU_UPLOAD), 정적 자원은 한 번 올려 상주.
- 통합 메모리: 이산은 UVM(자동 마이그레이션), APU는 물리 UMA(복사 없음).
