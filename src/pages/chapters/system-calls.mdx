# 시스템 콜과 user/kernel 경계

여러분이 짠 프로그램은 파일을 읽고, 메모리를 받고, 네트워크로 데이터를 보냅니다. 그런데 디스크 컨트롤러를
직접 건드리거나, 페이지 테이블을 직접 고치거나, 다른 프로세스의 메모리를 마음대로 읽을 수는 없습니다 —
*없어야* 합니다. 그렇게 할 수 있다면 버그 하나가 시스템 전체를 무너뜨리고, 악성 프로그램이 모든 것을
훔칠 테니까요.

그래서 CPU는 코드를 두 개의 권한 수준에서 실행합니다. 여러분의 코드는 권한이 낮은 곳(**ring 3**, user
mode)에서 돌고, 특권 명령과 하드웨어는 권한이 높은 곳(**ring 0**, kernel mode)에만 허락됩니다. 그런데
파일을 읽으려면 결국 디스크에 닿아야 합니다. 어떻게 권한의 벽을 넘어 커널에 일을 *부탁*할까요?

벽에는 단 하나의, 엄격히 통제된 문이 있습니다. 그 문이 **시스템 콜(system call)**입니다.

## 보호 링과 모드 전환

x86은 ring 0–3의 네 단계 특권을 정의하지만, 실제 OS는 보통 둘만 씁니다: 커널은 ring 0, 유저는 ring 3.
ring 3에서는 `cli`(인터럽트 끄기), I/O 포트 접근, 페이지 테이블 베이스(CR3) 쓰기 같은 특권 명령이
금지됩니다 — 시도하면 #GP 예외가 터집니다.

여기서 가장 흔한 오해를 짚고 갑시다. **유저가 직접 커널 함수로 점프할 수는 없습니다.** 커널 주소로
`call`이나 `jmp`를 해봐야, 권한이 낮은 코드가 권한이 높은 코드 영역을 실행하려는 것이므로 예외로
막힙니다. 권한 상승은 *주소로 점프하는* 방식으로는 불가능하고, **CPU가 제공하는 특정 트랩 명령을 통해
미리 정해진 진입점으로만** 일어납니다. 이 비대칭이 보안의 핵심입니다: 커널은 "어디로 들어올지"를 자기가
정하고, 유저는 그 문으로만 들어올 수 있습니다.

이 트랩 메커니즘은 [인터럽트와 예외](./interrupts-exceptions)에서 본 진입 경로와 본질적으로 같습니다.
인터럽트가 *바깥에서 비동기로* 흐름을 가로채는 것이라면, 시스템 콜은 유저가 *스스로, 동기적으로* 거는
트랩입니다 — 자발적인 예외라고 봐도 됩니다.

## 트랩 명령: int 0x80에서 syscall로

권한의 벽을 넘는 트랩 명령은 x86에서 세대를 거치며 발전했습니다.

- **`int 0x80` (레거시).** 가장 오래된 방식으로, 소프트웨어 인터럽트를 0x80번 벡터로 발생시킵니다. 즉
  앞 장의 IDT를 그대로 타며, `IDT[0x80]`의 게이트가 커널 진입점을 가리킵니다. 범용 인터럽트 경로를
  재사용하므로 유연하지만 *느립니다* — 인터럽트 진입의 모든 절차(권한 확인, 스택 전환 등)를 거칩니다.
- **`sysenter` (Intel) / `syscall` (AMD).** 시스템 콜만을 위한 전용 고속 명령입니다. IDT 조회 같은 범용
  절차를 건너뛰고, 미리 설정해 둔 MSR(레지스터)이 가리키는 커널 진입점으로 곧장 점프합니다. 32비트에서는
  주로 `sysenter`, 64비트에서는 `syscall`을 씁니다. 오늘날 64비트 Linux의 표준 경로가 `syscall`입니다.

Linux는 실행 중인 CPU가 무엇을 지원하는지 보고 셋 중 최적을 고릅니다(이를 vDSO의 `__kernel_vsyscall`이
감춰 줍니다). 우리는 이하 64비트 `syscall`을 기준으로 설명합니다.

## syscall ABI: 번호·인자·반환의 약속

커널은 수백 개의 시스템 콜을 제공합니다. 어떤 콜을 부르는지, 인자를 어떻게 넘기는지, 답을 어디서 받는지
— 이 약속이 **ABI(Application Binary Interface)**입니다. 함수 호출 규약과 비슷하지만 미묘하게 다릅니다.

<SyscallAbi client:visible />

규칙은 간결합니다. **번호**는 `rax`에, **인자**는 순서대로 `rdi, rsi, rdx, r10, r8, r9`에(최대 6개) 넣고
`syscall`을 실행합니다. **반환값**은 `rax`로 돌아옵니다. 한 가지 함정은 네 번째 인자입니다: 일반 함수 호출
규약(System V AMD64)은 `rcx`를 쓰지만, `syscall` 명령이 복귀 주소를 `rcx`에 저장하면서 덮어쓰기 때문에
시스템 콜은 네 번째 인자를 `r10`으로 옮깁니다(`r11`도 플래그 저장에 쓰여 망가집니다).

인자가 6개를 넘는 콜은 어떻게 할까요? 그럴 땐 인자들을 메모리의 구조체에 담고 그 *포인터*를 레지스터로
넘깁니다. 이것이 보안의 미묘한 지점입니다 — 유저가 넘긴 포인터는 커널이 **절대 그대로 믿으면 안 됩니다.**
커널이 검증 없이 유저 포인터를 따라가면 유저가 커널 메모리를 읽거나 쓰게 만들 수 있으니, 커널은 모든 유저
포인터를 `copy_from_user`/`copy_to_user`로 경계를 검사하며 복사합니다.

## 경계를 넘는 한 번의 왕복

이제 `syscall` 명령이 실제로 무엇을 하는지 한 단계씩 봅시다.

<SyscallPath client:visible />

`syscall`이 실행되면 하드웨어가 세 가지를 합니다: 복귀 주소(다음 명령의 RIP)를 `rcx`에, 현재 RFLAGS를
`r11`에 저장하고, `SFMASK` MSR이 지정한 플래그 비트를 끈 뒤(예: IF를 꺼 진입 중 인터럽트 차단), RIP를
`LSTAR` MSR이 담은 커널 진입점으로 바꿉니다. 동시에 특권이 ring 0으로 올라가고 CS/SS는 `STAR` MSR의
값으로 설정됩니다. 이 MSR들(LSTAR=진입 주소, STAR=세그먼트 셀렉터, SFMASK=플래그 마스크)은 부팅 시
커널이 `wrmsr`로 한 번 채워 둡니다.

여기서 인터럽트와의 결정적 차이가 있습니다. **`syscall`은 스택을 자동으로 전환하지 않습니다.** 인터럽트는
하드웨어가 커널 스택으로 바꿔 주지만, `syscall` 직후 RSP는 *여전히 유저 스택*을 가리킵니다. 그래서 커널
진입 코드는 가장 먼저 `swapgs`로 per-CPU 데이터의 베이스(GS)를 커널용으로 바꾸고, 그 per-CPU 영역에서
커널 스택 주소를 꺼내 RSP를 직접 갈아끼웁니다. 이 짧은 도입부가 끝나야 비로소 일반 C 커널 코드가 돌 수
있습니다.

그다음 커널은 `rax`의 번호로 시스템 콜 테이블을 찾아 해당 핸들러(`sys_read` 등)를 호출하고, 인자를
검증하고, 일을 수행합니다. 끝나면 `sysret` 명령이 `rcx`→RIP, `r11`→RFLAGS를 복원하며 특권을 ring 3로
내리고 유저 코드로 돌아옵니다.

## 비용: 왜 시스템 콜은 "비싼가"

함수 호출은 보통 몇 cycle이면 끝납니다. 시스템 콜은 그보다 훨씬 비쌉니다. 작업 자체가 사소한
콜(`getpid()`처럼 값 하나 반환)조차 명령의 진입·복귀만으로 현대 CPU에서 대략 **수십~100 cycle** 정도가
듭니다(마이크로아키텍처 의존, 자릿수로 보세요). 이 비용은 어디서 올까요?

- **모드 전환 자체.** 특권 레벨 변경, 세그먼트·스택 갱신, 플래그 마스킹.
- **간접 비용.** 커널 진입 코드와 핸들러가 명령·데이터 캐시와 분기 예측기를 유저 코드의 것으로부터 밀어내
(오염), 복귀 후 유저 코드가 다시 캐시 미스를 겪습니다. 이 *간접* 비용이 종종 직접 비용보다 큽니다.
- **KPTI(Meltdown 완화).** Meltdown 취약점을 막기 위해 유저/커널의 페이지 테이블을 분리하면, 모드 전환마다
  페이지 테이블을 갈아끼우고 TLB를 비워야 합니다. syscall당 대략 30–70 cycle이 추가되고, 시스템 콜이
  *아주 잦은* 워크로드에서는 20–30%까지 느려질 수 있습니다(일반 워크로드에서는 훨씬 작습니다). 하드웨어가
  PCID를 지원하면 TLB를 통째로 비우지 않아도 돼 부담이 줄어듭니다.

페이지 테이블과 TLB가 왜 모드 전환 비용을 좌우하는지는 [가상 메모리와 TLB](./virtual-memory-tlb)에서
자세히 다룹니다. 요점은 명확합니다: 시스템 콜을 *자주* 부르는 것은 비싸고, 그래서 가능하면 피하거나
묶어야 합니다.

## vDSO: 경계를 안 넘는 빠른 경로

비용이 진입·복귀에서 온다면, 가장 빠른 시스템 콜은 *시스템 콜을 하지 않는 것*입니다. 일부 콜은 커널의
특권이 전혀 필요 없습니다 — 예컨대 "지금 몇 시야?"(`clock_gettime`)는 *읽기 전용*이고, 답을 만드는 데
필요한 데이터를 유저가 봐도 무방합니다.

<VdsoFastPath client:visible />

그래서 커널은 **vDSO(virtual Dynamic Shared Object)**를 둡니다. 이것은 커널이 *모든 프로세스의
주소공간에* 매핑해 주는 작은 공유 라이브러리입니다(매핑 위치는 보안을 위해 ASLR로 무작위화됩니다). vDSO
안의 코드는 커널이 비동기로 갱신하는 공유 메모리 페이지에서 시각 기준값을 읽고, CPU의 타임스탬프 카운터
(`rdtsc`)로 보정해 현재 시각을 *유저 모드에서* 계산합니다. **경계를 한 번도 넘지 않으므로** 진입·복귀
비용이 통째로 사라집니다.

vDSO가 처리하는 콜은 아키텍처·커널 버전에 따라 다르지만 대표적으로 `clock_gettime`, `gettimeofday`,
`time`, `getcpu` 같은 read-only 콜입니다. 반대로 상태를 *바꾸는* 콜(`write`, `open`, `mmap` …)은 반드시
커널의 특권이 필요하므로 vDSO로 만들 수 없고, 진짜 `syscall` 경로를 타야 합니다. 좋은 설계는 "이 작업에
정말 커널이 필요한가"를 따져, 필요 없는 것은 경계 밖으로 빼는 것입니다.

## libc 래퍼와 errno

응용 코드가 `syscall` 명령을 직접 쓰는 일은 거의 없습니다. 대신 libc의 래퍼 함수(`read()`, `open()` …)를
부르고, 래퍼가 번호와 레지스터를 세팅해 명령을 실행합니다. 래퍼는 두 가지를 더 해 줍니다.

첫째, 위에서 본 vDSO 빠른 경로를 *자동으로* 골라 줍니다 — 여러분이 `clock_gettime`을 불러도 libc가 내부
적으로 vDSO 버전이 있으면 그쪽을 씁니다.

둘째, **errno 변환**입니다. 커널은 성공하면 0 이상, 실패하면 작은 *음수* 에러 코드를 `rax`에 담아 돌려
줍니다(관례상 $-4095 \le \text{ret} \le -1$ 범위의 음수가 에러 — `MAX_ERRNO`=4095). 그런데 C 프로그래머는 음수 반환을
직접 검사하기보다 "반환 -1, 에러 코드는 `errno`에"라는 관례에 익숙합니다. 그래서 래퍼는 반환값이 에러
범위의 음수면 그 절댓값을 전역(스레드별) `errno`에 넣고 -1을 반환합니다:

$$
\text{ret} < 0 \;\Rightarrow\; \texttt{errno} = -\text{ret}, \quad \text{반환값} = -1.
$$

즉 `errno`는 커널이 아니라 *libc가* 만들어 내는 사용자 편의 계층입니다 — 시스템 콜의 raw 반환값과 응용
코드의 익숙한 에러 처리 사이의 얇은 어댑터입니다.

## 조립: read() 한 번의 전체 경로

이제 `read(fd, buf, n)` 한 줄이 무슨 일을 일으키는지 끝까지 따라갈 수 있습니다.

1. 응용 코드가 libc의 `read()` 래퍼를 부릅니다.
2. 래퍼가 `rax`=read의 번호, `rdi`=fd, `rsi`=buf, `rdx`=n을 세팅하고 `syscall`을 실행합니다.
3. 하드웨어가 `rcx`←복귀 RIP, `r11`←RFLAGS를 저장하고, ring 0으로 올라가 `LSTAR`의 커널 진입점으로
   점프합니다.
4. 진입 코드가 `swapgs` 후 커널 스택으로 전환하고, 범용 레지스터를 저장합니다.
5. 커널이 번호로 `sys_read`를 찾아 호출하고, `buf` 포인터의 유효성을 검사한 뒤 데이터를 복사합니다(필요
   하면 디스크 I/O를 시작하고 — 그 완료는 [인터럽트](./interrupts-exceptions)로 통지받습니다).
6. `sysret`이 `rcx`/`r11`에서 RIP/RFLAGS를 복원해 ring 3로 돌아옵니다.
7. 래퍼가 `rax`를 검사해, 음수면 `errno`를 채우고 -1을, 아니면 읽은 바이트 수를 반환합니다.

한 줄의 코드 아래에 권한 경계의 왕복, 두 개의 MSR, 스택 전환, 포인터 검증, 그리고 errno 어댑터가
숨어 있습니다. 시스템 콜이 "그냥 함수 호출보다 비싼" 이유가 여기 다 들어 있습니다.

## 더 나아가기

- **시스템 콜 배칭.** 콜마다 경계를 넘는 게 비싸니, 여러 작업을 한 번에 제출하는 인터페이스가 등장했습니다 —
  `io_uring`은 유저/커널이 공유하는 링 버퍼로 다수의 I/O를 경계 왕복 없이 제출·완료합니다. vDSO가 *읽기*를
  경계 밖으로 뺐다면, io_uring은 *제출*을 묶어 경계 횟수를 줄입니다.
- **seccomp.** 커널은 프로세스가 부를 수 있는 시스템 콜을 필터링할 수 있습니다(샌드박싱). 문을 좁히는
  보안 계층입니다.
- **다른 OS·ISA.** 번호와 레지스터 매핑은 OS·아키텍처마다 다릅니다(Windows는 비공개·버전마다 변동, ARM64는
  `svc` 명령에 `x8`=번호·`x0–x5`=인자). 메커니즘(트랩→진입점→핸들러→복귀)은 이식되지만 ABI 숫자는 다릅니다.
- **모드 전환 없는 통신.** 공유 메모리와 vDSO처럼, 경계 비용을 줄이는 방향은 계속 발전 중입니다 — 핵심
  통찰은 한결같습니다: *경계를 넘는 일을 줄여라.*
