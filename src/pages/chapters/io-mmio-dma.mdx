# 장치 I/O — MMIO·포트 I/O·DMA

CPU의 명령 집합에는 `add`, `load`, `store`, `jmp`는 있어도 "디스크에서 1 MB 읽어라"나 "이 픽셀들을
화면에 보내라" 같은 명령은 없습니다. CPU가 아는 것은 레지스터와 메모리뿐입니다. 그런데도 컴퓨터는
디스크를 읽고, 네트워크를 주고받고, GPU에 프레임을 보냅니다. 어떻게? **장치를 메모리처럼 보이게**
만들고, 무거운 전송은 **CPU를 아예 빼버리는** 방식으로 합니다. 이 챕터는 그 두 트릭 — 장치 레지스터에
닿는 법(MMIO·port I/O)과 데이터를 옮기는 법(programmed I/O·인터럽트·DMA) — 을 다룹니다.

## 장치는 레지스터 묶음이다

소프트웨어 관점에서 하나의 장치는 몇 개의 **하드웨어 레지스터**입니다. 종류는 대체로 셋입니다.

- **status 레지스터**: 장치가 지금 바쁜지, 데이터가 준비됐는지, 에러가 났는지(읽기 전용인 경우가 많음).
- **command 레지스터**: 장치에게 "읽기 시작", "리셋" 같은 동작을 시킴(쓰기).
- **data 레지스터**: 실제로 주고받는 바이트가 드나드는 창구.

장치를 제어한다는 것은 결국 이 레지스터들을 정해진 순서로 읽고 쓰는 일입니다. 예를 들어 "command에
읽기 명령을 쓰고 → status의 ready 비트가 설 때까지 기다렸다가 → data에서 한 워드를 읽는다"처럼. 그러면
CPU는 이 레지스터들을 *어떤 주소로* 접근할까요? 여기서 두 갈래가 갈립니다.

## MMIO vs port I/O — 레지스터에 닿는 두 길

<AddressSpaceMap client:visible />

**memory-mapped I/O(MMIO)**는 장치 레지스터를 CPU의 **물리 주소 공간** 한가운데에 끼워 넣습니다. 그러면
장치 레지스터가 메모리의 한 주소처럼 보이므로, CPU는 특별한 명령 없이 평범한 load/store(x86이면 `MOV`)로
장치를 읽고 씁니다. 장치들은 주소 버스를 감시하다가 자기에게 할당된 주소 범위로의 접근에 반응합니다.
PCI/PCIe 장치는 이 범위를 **BAR(base address register)**로 노출하고, 펌웨어·OS가 부팅 때 BAR에 물리
주소를 배정합니다.

MMIO에는 반드시 지켜야 할 제약이 하나 있습니다 — 이 영역은 **캐시 불가(uncacheable)**여야 합니다. 보통
메모리는 같은 주소를 다시 읽으면 같은 값이 나오고, 그래서 캐시가 안전합니다. 하지만 장치 레지스터는
다릅니다. status를 읽을 때마다 값이 바뀔 수 있고(장치가 바꿈), data 레지스터는 읽는 행위 자체가 다음
바이트를 당겨오는 **부수효과**를 가질 수 있습니다. 이런 주소를 캐시에 들고 있으면 CPU는 옛 값을 보거나
부수효과를 건너뜁니다. 그래서 MMU/MTRR로 이 영역을 uncacheable로 표시합니다(VGA 프레임버퍼처럼 캐시를
허용하는 특수 예외는 있습니다).

**port-mapped I/O(PMIO 또는 port I/O)**는 x86의 오래된 방식으로, 메모리와 **완전히 분리된** I/O 주소
공간을 둡니다. x86에는 0부터 0xFFFF까지 총 65536개(64K)의 I/O port가 있고, 여기엔 평범한 `MOV`가 아니라
전용 명령 `IN`/`OUT`으로만 접근합니다. `IN`/`OUT`은 `EAX` 레지스터와 지정한 port 사이로 1·2·4바이트를
옮깁니다. 키보드 컨트롤러(0x60), 레거시 디스크(0x1F0), 시리얼 포트(0x3F8), PCI 설정 공간 접근
창구(0xCF8/0xCFC) 같은 것이 port로 노출됩니다.

두 공간이 별개라는 점이 중요합니다 — 같은 숫자 "0x60"이라도 물리 메모리 주소로서의 0x60(RAM의 한
바이트)과 I/O port 0x60(키보드)은 전혀 다른 대상입니다. 어느 명령(`MOV` vs `IN`/`OUT`)을 쓰느냐가 어느
공간이냐를 결정합니다.

오늘날 새 장치는 거의 다 MMIO를 씁니다. 64K port는 비좁고, MMIO는 일반 메모리 명령을 그대로 써서
주소 폭도 넓고 코드도 단순합니다. 무엇보다 ARM 등 많은 아키텍처엔 애초에 port 공간이라는 개념이 없어
**모든 장치가 MMIO**입니다. port I/O는 x86의 역사적 유산에 가깝습니다.

## 데이터를 옮기는 법 (1) — programmed I/O와 폴링

레지스터에 닿는 길을 정했으니, 이제 실제로 데이터 블록을 옮길 차례입니다. 가장 단순한 방식은 **programmed
I/O(PIO)**입니다 — CPU가 직접, 한 워드씩, 자기 손으로 옮깁니다. 디스크에서 한 섹터를 읽는다면 대략 이런
루프입니다.

1. command 레지스터에 "읽기" 명령을 쓴다.
2. status 레지스터를 **계속 읽으며** ready 비트가 설 때까지 기다린다(이게 **폴링/busy-wait**).
3. 준비되면 data 레지스터에서 한 워드를 읽어 메모리에 저장한다.
4. 블록을 다 옮길 때까지 2–3을 반복한다.

문제는 2번입니다. CPU가 status를 끝없이 읽으며 기다리는 동안 — 그리고 워드를 하나씩 직접 나르는 동안 —
CPU는 그 일에 통째로 묶여 다른 아무것도 못 합니다. 디스크는 CPU보다 수만 배 느리니, 이 폴링은 대부분
"아직?"을 반복하며 사이클을 태우는 낭비입니다.

## 데이터를 옮기는 법 (2) — 인터럽트

폴링의 첫 번째 개선은 **인터럽트**입니다. status를 계속 읽는 대신, CPU는 command를 보내고 곧바로 다른
일을 하러 갑니다. 장치는 준비가 되면 **interrupt request(IRQ)** 신호로 CPU를 부르고, CPU는 하던 일을
잠시 멈추고 인터럽트 핸들러(ISR)로 점프해 데이터를 처리한 뒤 돌아옵니다. 이제 CPU는 기다리는 시간을
낭비하지 않습니다.

하지만 인터럽트는 *기다림*만 없앴을 뿐, **바이트를 나르는 일은 여전히 CPU 몫**입니다. 1 MB를 옮기면
data 레지스터를 통한 워드 단위 복사가 그만큼 일어나고, 빠른 장치(고속 NIC·NVMe)에선 인터럽트가 너무
자주 떠서 핸들러 진입·복귀 오버헤드만으로도 CPU가 휘청입니다. 큰 전송에는 더 근본적인 방법이 필요합니다.
(인터럽트 자체의 메커니즘 — 벡터 테이블, 우선순위, 예외와의 관계 — 은 [`./interrupts-exceptions`](./interrupts-exceptions)에서
다룹니다.)

## 데이터를 옮기는 법 (3) — DMA

**DMA(direct memory access)**의 발상은 단순합니다 — **바이트를 나르는 일에서 CPU를 빼버린다.** CPU는
전송의 *조건*만 정해주고, 실제 메모리↔장치 전송은 **DMA 컨트롤러**(또는 그 기능을 내장한 장치)가 직접
합니다.

<PioVsDma client:visible />

PIO에서는 CPU가 전송 내내 묶여 있던 반면, DMA에서는 CPU가 설정만 짧게 하고 곧 풀려나 다른 일을 합니다.
실제 바이트 이동은 DMA 컨트롤러가 메모리와 장치 사이에서 처리하고, 다 끝나면 **완료 인터럽트**를 한 번만
올립니다. 전송이 클수록 이득이 커집니다 — 그래서 디스크·네트워크·GPU 전송은 사실상 전부 DMA입니다.

여기서 "DMA 컨트롤러"의 의미는 시대에 따라 다릅니다. 옛 ISA 시절엔 메인보드에 중앙 DMA 컨트롤러(8237
같은)가 있어 장치들이 그것을 빌려 썼습니다. 현대 PCIe에서는 별도 중앙 컨트롤러가 아니라 **장치 자신이
bus master가 되어** 직접 PCIe 트랜잭션(memory read/write TLP)을 발생시켜 host RAM에 접근합니다. NIC·NVMe·GPU가
모두 이 방식입니다. 큰 전송은 보통 흩어진 메모리 버퍼들을 가리키는 **descriptor 리스트**(scatter-gather)로
묶어, 컨트롤러가 리스트를 따라 자율적으로 옮기고 리스트 끝에서만 인터럽트를 올립니다.

### DMA의 두 함정 — 캐시 일관성과 보호

DMA가 RAM을 CPU 우회로 직접 건드린다는 바로 그 장점이 두 가지 문제를 부릅니다.

**캐시 일관성.** DMA 컨트롤러가 RAM에 데이터를 직접 써넣었는데, CPU 캐시에 그 주소의 *옛* 사본이 남아
있으면, CPU는 새 데이터 대신 stale 값을 봅니다. 반대로 CPU가 캐시에만 써둔(아직 RAM에 안 내려간) 데이터를
DMA가 RAM에서 읽어가면 옛 값을 가져갑니다. 플랫폼이 하드웨어로 일관성을 보장(cache-coherent)하면 신경
쓸 게 없지만 — x86은 대체로 그렇습니다 — 많은 ARM SoC처럼 비coherent 플랫폼에서는 드라이버가 DMA 전후로
해당 영역을 직접 **flush/invalidate**해야 합니다. 이것이 OS의 DMA 매핑 API(`dma_map_*` 등)가 존재하는
이유 중 하나입니다.

**보호.** DMA는 물리 주소로 직접 메모리를 건드리므로, 잘못된(혹은 악의적인) 장치가 임의의 물리
메모리를 읽고 쓸 수 있습니다. 이를 막는 것이 **IOMMU**입니다 — CPU의 MMU가 프로세스를 위해 가상→물리
변환을 하듯, IOMMU는 장치가 내는 주소(I/O 가상 주소)를 물리 주소로 변환하며, 장치가 허가된 영역 밖을
못 건드리게 격리합니다(상세는 이 챕터 범위 밖).

## 종합

CPU가 아는 것은 레지스터와 메모리뿐이라는 한 가지 사실에서 전부가 따라 나왔습니다.

- 장치를 다루려면 그 **레지스터**(status·command·data)에 닿아야 하고, 그 길은 **MMIO**(물리 주소 공간에
  매핑, 일반 load/store, 단 uncacheable) 또는 x86의 **port I/O**(별도 64K 공간, `IN`/`OUT`)입니다.
- 데이터를 옮기는 방식은 **PIO 폴링**(CPU가 직접·busy-wait) → **인터럽트**(기다림 제거, 그러나 복사는
  여전히 CPU) → **DMA**(복사까지 컨트롤러에 위임, CPU는 설정과 완료 인터럽트만)로 발전합니다.
- DMA의 직접 접근이라는 장점은 **캐시 일관성**과 **보호(IOMMU)**라는 두 숙제를 동반합니다.
