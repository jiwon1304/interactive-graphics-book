# CPU 캐시와 메모리 계층

## 같은 배열, 같은 합 — 왜 한쪽이 6배 느릴까?

정사각 행렬 하나를 두 가지 순서로 더해 봅시다. 첫 번째는 행을 따라(같은 행 안에서 옆으로),
두 번째는 열을 따라(같은 열 안에서 아래로) 같은 원소를 전부 더합니다.

```c
// (A) 행 우선 — 안쪽 루프가 열 인덱스
for (int i = 0; i < N; i++)
  for (int j = 0; j < N; j++) sum += a[i][j];

// (B) 열 우선 — 안쪽 루프가 행 인덱스
for (int j = 0; j < N; j++)
  for (int i = 0; i < N; i++) sum += a[i][j];
```

두 루프는 **정확히 같은 원소를 같은 횟수만큼** 더합니다. 산술 연산 수도, 결과도 같습니다.
그런데 큰 `N`에서 (B)는 (A)보다 몇 배씩 느립니다. CPU는 같은 덧셈을 하고 있는데
무엇이 다른 걸까요?

답은 연산이 아니라 **데이터가 코어에 도착하는 데 걸리는 시간**에 있습니다. 이 챕터는 그
시간이 왜 천차만별인지 — 그리고 코드가 그 시간을 어떻게 좌우하는지를 다룹니다.

## memory wall — 느린 건 계산이 아니라 메모리다

수십 년간 CPU의 연산 속도와 DRAM의 접근 속도는 **다른 속도로** 빨라졌습니다. 1995년
Wulf와 McKee는 이 격차가 두 지수함수의 차이라서 *기하급수적으로* 벌어진다고 지적했습니다 —
이것이 **"memory wall"** 입니다. 흔히 인용되는 추세값으로 프로세서 성능은 연 ~50–60%,
DRAM 지연은 연 ~7% 개선되었다고 하는데, 이 백분율은 Wulf–McKee 본문의 인용이라기보다
Hennessy & Patterson의 추세 데이터로 굳어진 **대표값**입니다. 방향은 분명합니다: 코어는
점점 빨라지는데 메모리는 거의 그 자리에 있었습니다.

그 결과 오늘날 코어 하나가 DRAM에서 데이터 한 덩어리를 기다리는 동안, 이론상 수백 개의
산술 명령을 놀린 채 흘려보낼 수 있습니다. 대략적인 지연을 cycle 단위로 보면(Intel Skylake
측정값, 7-cpu.com — Agner Fog 표로 교차확인한 **대표값**):

이 그림이 이 챕터의 출발점입니다. 한 칸 내려갈 때마다 지연이 대략 한 자릿수씩 커집니다.
그러니 해법은 자명합니다 — **자주 쓰는 데이터를 위쪽(빠른 층)에 붙들어 둔다.** 그 "붙들어
두는" 자동 장치가 바로 **캐시(cache)** 입니다.

> **두 독자를 위한 메모.** cycle vs ns: cycle 수는 클럭에 무관한 상대 비용이라 비교에 좋지만,
> DRAM 같은 고정 지연(~50–100 ns)은 클럭이 빨라질수록 *cycle 수가 늘어납니다*. 그래서
> "L1 ≈ 4 cycle, DRAM ≈ 200+ cycle"은 마이크로아키텍처를 귀속해 읽어야 합니다. ns로 외우는
> 고전 표(Jeff Dean/Norvig: L1 ≈ 0.5 ns, L2 ≈ 7 ns, 메모리 ≈ 100 ns)도 자릿수 감각용입니다.

## 왜 캐시가 통하는가 — 지역성(locality)

캐시가 작동하는 이유는 프로그램의 메모리 접근이 무작위가 아니기 때문입니다. 두 가지
경향(**principle of locality**)이 있습니다(Hennessy & Patterson / CSAPP §6.2):

- **temporal locality(시간 지역성):** 방금 접근한 주소는 곧 또 접근될 가능성이 높다.
  (루프 변수, 자주 호출되는 함수.)
- **spatial locality(공간 지역성):** 어떤 주소를 접근했다면 그 *근처* 주소도 곧 접근될
  가능성이 높다. (배열을 순서대로 훑기, 구조체 필드 연달아 읽기.)

캐시는 이 두 경향에 각각 베팅합니다. temporal locality에는 "한 번 가져온 데이터를 잠시
보관"으로, spatial locality에는 **byte가 아니라 한 덩어리(cache line)를 통째로** 가져오는
방식으로 대응합니다.

## 캐시라인 — 메모리는 64 byte 묶음으로 움직인다

CPU는 메모리를 1 byte씩 가져오지 않습니다. 항상 정렬된 **캐시라인(cache line)** 단위로
가져옵니다. x86-64에서는 거의 예외 없이 **64 byte**입니다(Intel·AMD 모두). ARM은
아키텍처가 강제하지 않아 구현마다 다르지만 최근 코어는 대개 64 B입니다. 잘 알려진 예외는
**Apple M 계열로 128 B** 라인을 씁니다(7-cpu.com 측정).

> **두 독자를 위한 메모.** 64 B를 코드에 박지 마세요. C++이라면
> `std::hardware_destructive_interference_size`나 런타임 질의로 라인 크기를 얻는 편이
> 안전합니다(Apple Silicon 128 B 때문에).

이 "라인 단위"가 만드는 직관에 어긋나는 효과가 있습니다. `int`(4 B) 배열을 stride(건너뛰기
간격)를 바꿔 가며 더한다고 합시다. 한 라인 64 B = `int` 16개입니다.

<CacheLineStride client:visible />

stride를 1, 2, 4, 8, 16으로 키워도 — 같은 라인들을 모두 한 번씩은 건드리는 한 —
**캐시 미스 수는 똑같습니다.** 미스는 byte당이 아니라 *라인당* 일어나기 때문입니다. 그래서
실행 시간이 거의 변하지 않습니다. stride가 16을 **넘어서야** 라인을 통째로 건너뛰기
시작하고, 그제서야 빨라집니다(Ostrovsky, "Gallery of Processor Cache Effects", 예제 1).

이제 맨 앞의 행렬 수수께끼가 풀립니다. C는 행 우선(row-major) 저장이라 한 행의 원소가
메모리에서 연속입니다. **(A) 행을 따라** 훑으면 한 라인(원소 ~16개)을 가져와 16번 다
쓰고 다음 라인으로 갑니다 — spatial locality 만점입니다. **(B) 열을 따라** 훑으면 라인마다
원소 1개만 쓰고 나머지 15개는 버린 채 다음 라인으로 건너뜁니다. 같은 덧셈인데 메모리
트래픽은 십수 배입니다(CSAPP의 고전 예제; Drepper §6). 느린 건 계산이 아니라 *낭비된
라인*입니다.

## 캐시는 데이터를 어디에 두는가 — set-associative 구조

캐시는 가져온 라인을 아무 데나 두지 않습니다. 빠르게 찾으려면 "이 주소의 라인이 있다면
*여기*에만 있을 수 있다"는 규칙이 필요합니다. 그래서 하드웨어는 주소를 세 토막으로 봅니다.

라인 크기가 $B = 2^b$ byte, set이 $S = 2^s$ 개, 각 set에 라인을 둘 자리(way)가 $E$ 개라고
합시다. 주소는 상위 비트부터 **tag · index · offset** 으로 쪼개집니다:

$$
\underbrace{\text{tag}}_{t\ \text{bit}}\ \Big|\ \underbrace{\text{index}}_{s\ \text{bit}}\ \Big|\ \underbrace{\text{offset}}_{b\ \text{bit}}
$$

- **offset** $b = \log_2 B$ : 라인 *안에서* 몇 번째 byte인가.
- **index** $s = \log_2 S$ : 어느 **set**에 속하는가.
- **tag** $t = (\text{주소 폭}) - s - b$ : 그 set 안에서 어느 라인인가(나머지 상위 비트).

set 안 자리 수 $E$ 가 매핑 방식을 정합니다: $E=1$ 이면 **direct-mapped**(라인이 갈 곳이
딱 한 군데), $E>1$ 이면 **$E$-way set-associative**, set이 하나뿐($S=1$)이면 **fully
associative**(아무 데나). 실제 코어는 그 사이를 택합니다 — 예를 들어 최근 Intel
Golden Cove의 L1d는 48 KB · 12-way, L2는 16-way입니다(chipsandcheese / wikipedia).

<AddressDecomposition client:visible />

set 수와 way 수, 라인 크기는 서로 묶여 있습니다. 위 도식의 예시(32 KB · 8-way · 64 B)로
유도해 봅시다. 전체 라인 수는 $32768 / 64 = 512$, 8-way이므로 set 수는
$512 / 8 = 64 = 2^6$ → index 6 bit. offset은 $\log_2 64 = 6$ bit. 일반식:

$$
S = \frac{\text{캐시 크기}}{B \cdot E}, \qquad s = \log_2 S
$$

조회는 이렇게 진행됩니다: index로 **set 하나**를 고르고, 그 set 안 $E$ 개 way의 tag를
**병렬로 동시에** 비교합니다. 하나라도 맞고 유효하면 **hit**, 그 라인의 offset 위치에서
데이터를 꺼냅니다. 모두 어긋나면 **miss**, 아래 층에서 라인을 가져와 이 set의 한 자리에
채웁니다.

## 어느 자리를 비울까 — 교체 정책

set이 꽉 찼는데 새 라인이 오면 하나를 쫓아내야 합니다. 교과서는 흔히 가장 오래 안 쓴
라인을 버리는 **LRU(Least Recently Used)** 를 가르치지만, 실제 고연관(high-associativity)
캐시에서 *진짜* LRU는 비용이 커서 잘 안 씁니다. 현실의 코어는 **pseudo-LRU(tree-PLRU)** 나
더 정교한 적응형 정책(예: Intel의 일부 L3는 set-dueling 기반 adaptive replacement,
학술적으로는 RRIP 계열)을 씁니다. Intel은 정확한 정책을 공개하지 않아 대부분 리버스
엔지니어링으로 알려진 내용입니다 — 그러니 "LRU에 가깝다" 정도로 이해하세요.

## 미스는 왜 나는가 — 세 가지 C

캐시 미스의 원인은 셋으로 분류됩니다(Hill & Smith, H&P로 대중화 — **3C**):

- **compulsory(cold) miss:** 그 라인을 *처음* 만질 때. 캐시가 아무리 커도 첫 접근은 미스다.
- **capacity miss:** working set이 캐시보다 커서, 쫓겨난 라인을 나중에 다시 가져와야 한다.
- **conflict miss:** 캐시 전체엔 자리가 있는데 **특정 set에만 라인이 몰려** 서로를 쫓아낸다.
  set-associative·direct-mapped에서만 생긴다(fully associative엔 없음).

평균 접근 시간을 한 식으로 묶으면 **AMAT(Average Memory Access Time)** 입니다(H&P / CSAPP):

$$
\text{AMAT} = t_{\text{hit}} + \text{(miss rate)} \times \text{(miss penalty)}
$$

다층 캐시에서는 재귀적으로 확장됩니다 — L1의 miss penalty 자리에 L2의 AMAT가 들어가고,
그 안에 다시 L3의 AMAT가… 식으로요. 이 식이 말하는 바는 단순합니다: hit를 빠르게 하거나,
miss rate를 낮추거나, miss penalty를 줄여라. 지역성을 살린 코드는 가운데 항(miss rate)을
직접 공격합니다.

> **두 독자를 위한 메모.** "캐시는 latency를 줄인다"가 1차 효과지만, 한 번 가져온 라인을
> 여러 번 재사용하면 *유효 대역폭*도 늘어납니다. working set을 키워 가며 원소당 접근 비용을
> 재면 L1→L2→L3→DRAM 경계마다 한 단씩 올라가는 **계단(staircase)** 이 나옵니다 — Drepper의
> 측정과 lmbench `lat_mem_rd`의 전형적 결과입니다.

## 2의 거듭제곱이 부르는 절벽 — conflict의 실제

index 비트는 주소의 *중간* 비트입니다. 그래서 stride가 **큰 2의 거듭제곱**(예: 256,
512, 4096 byte)이면, 연속으로 접근하는 주소들의 index가 **전부 같아집니다.** 모든 접근이
같은 set으로 몰리는 것이죠. 그 set의 way 수(예: 8 또는 16)를 넘는 순간, 라인들이 서로를
계속 쫓아내며 매 접근이 미스가 됩니다 — working set이 캐시 전체로는 한참 작은데도요.

그래서 메모리 벤치마크에서 stride를 키우면 성능이 매끄럽게 변하지 않고, 256·512·4096
같은 지점에서 **절벽**이 나타납니다(Ostrovsky, 예제 4). 큰 2D 배열의 행 길이를 정확히
$2^k$로 잡았다가 열 방향 접근이 한 set에 몰려 느려지는 함정도 같은 원리입니다 — 그래서
때로 행에 한 줄을 **패딩**해 2의 거듭제곱을 깨뜨립니다.

이 그림이 [`./gpu-memory-hierarchy`](./gpu-memory-hierarchy)의 GPU 공유 메모리 **bank
conflict**와 같은 가족입니다 — 둘 다 "주소의 중간 비트가 같으면 한 자원에 몰린다"는 이야기죠.

## 왜 순차 접근은 더 빠른가 — 하드웨어 prefetcher

지금까지는 "순차 접근은 라인을 알뜰히 쓴다"였는데, 사실 한 가지가 더 있습니다. 코어에는
**하드웨어 prefetcher** 가 있어 접근 패턴을 관찰하고 다음에 필요할 라인을 *미리* 가져옵니다.
최근 Intel 코어는 여러 prefetcher를 둡니다 — L1 next-line, L1 IP 기반 stride(명령별 일정한
보폭을 추적), L2 adjacent-line, L2 streamer(오름/내림 스트림 감지). 덕분에 순차·일정 stride
접근은 미스 페널티가 상당 부분 *숨겨집니다*(Intel 최적화 매뉴얼). 다만 L2 streamer는 보통
4 KB **페이지 경계를 넘지 못하므로**, 패턴이 페이지를 가로지르면 효과가 끊깁니다.

## 코어가 여럿일 때 — false sharing

지금까지는 코어 하나 이야기였습니다. 코어가 여럿이고 각자 캐시를 가지면, 같은 라인의 사본이
여러 캐시에 생길 수 있습니다. 이때 한 코어가 라인에 **쓰면** 코히런시 프로토콜(MESI 등)이
다른 코어의 사본을 무효화(invalidate)해야 데이터가 어긋나지 않습니다.

여기서 고약한 함정이 나옵니다. 두 스레드가 **서로 다른 변수**를 쓰는데, 그 변수들이
*우연히 같은 64 B 라인*에 들어 있다면? 데이터상 아무 의존이 없는데도, 한쪽이 쓸 때마다
상대 코어의 라인이 무효화되어 라인이 코어 사이를 핑퐁처럼 오갑니다 — 이것이 **false
sharing(거짓 공유)** 입니다. 흔한 예가 스레드별 카운터를 배열에 다닥다닥 붙여 둔 경우입니다.
해법은 각 변수를 캐시라인 경계에 맞추는 것(`alignas(64)`)으로, 라인을 갈라 놓는 겁니다
(Oracle "What Is False Sharing"; Ostrovsky 예제 6).

> **두 독자를 위한 메모.** false sharing의 "왜 무효화되는가"는 **MESI** 상태기계가 답합니다 —
> 한 라인이 Modified/Exclusive/Shared/Invalid 중 어디에 있는지로 코어 간 일관성을 지키죠.
> 이건 다음 시리즈 [`./memory-consistency-mesi`](./memory-consistency-mesi)에서 상태기계로
> 풀어냅니다.

## 종합 — 코드가 캐시에게 하는 부탁

같은 결과를 내는 두 코드의 속도가 갈리는 건 거의 언제나 *메모리를 어떻게 건드리느냐*
때문입니다. 이 챕터의 도구로 요약하면:

- 데이터를 **순서대로**(spatial locality) 훑어 라인을 알뜰히 쓰고 prefetcher를 돕는다.
- 자주 쓰는 것을 **반복해서 가까이**(temporal locality) 두어 위쪽 층에 머물게 한다.
- 큰 **2의 거듭제곱 stride**를 피하거나 패딩으로 깨 conflict 절벽을 비킨다.
- 멀티스레드에선 자주 쓰는 변수를 **라인 단위로 분리**해 false sharing을 막는다.

전부 같은 한 문장의 변주입니다 — *데이터를 코어 가까이, 라인을 통째로 알뜰히 써라.*

## 더 나아가기

- **inclusivity:** 전통적 Intel L3는 inclusive(상위 캐시 내용을 L3가 포함)였지만
  Skylake-server부터 non-inclusive로 바뀌었고, AMD Zen의 L3는 victim cache(거의 exclusive)
  입니다 — 일반화는 조심.
- **write 정책:** hit 시 write-through vs write-back(dirty 비트), miss 시 write-allocate vs
  no-write-allocate. 흔한 조합은 write-back + write-allocate.
- **가상 메모리·TLB:** 캐시가 *데이터*를 캐싱하듯 TLB는 *주소 번역*을 캐싱합니다. 큰 working
  set은 캐시뿐 아니라 TLB도 thrash합니다 → [`./virtual-memory-tlb`](./virtual-memory-tlb).
- **코어 간 일관성:** false sharing의 배경인 MESI와 메모리 정렬(x86 TSO vs ARM weak) →
  [`./memory-consistency-mesi`](./memory-consistency-mesi).
- GPU 쪽의 같은 주제(메모리 합치기·bank conflict)는
  [`./gpu-memory-hierarchy`](./gpu-memory-hierarchy)에서 다룹니다.
