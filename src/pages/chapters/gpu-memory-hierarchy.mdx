# GPU 메모리 계층과 합치기

[메모리 대역폭과 Roofline](./memory-bandwidth-roofline) 챕터에서 "현대 GPU의 진짜 병목은 대역폭"임을
봤습니다. 그런데 같은 하드웨어, 같은 데이터 양인데도 **접근 방식에 따라 유효 대역폭이 수 배에서
수십 배까지 달라집니다.** 이 챕터는 그 이유 — GPU의 메모리 계층과, 워프의 접근을 최소한의
트랜잭션으로 묶는 **합치기(coalescing)**, 그리고 공유 메모리의 **뱅크 충돌** — 을 다룹니다.

## 메모리 계층 — 빠르고 작은 것부터 느리고 큰 것까지

CPU와 마찬가지로 GPU도 위로 갈수록 빠르고 작아지는 여러 단계의 메모리 계층을 가집니다. 다만 GPU는 **수만 개의 스레드를 동시에** 굴리려고
온칩 자원을 극단적으로 키웠습니다. NVIDIA SM(스트리밍 멀티프로세서) 하나 기준으로(아키텍처별로
수치가 다르니 세대를 함께 적습니다):

- **레지스터 파일**: SM당 **65,536개의 32-bit 레지스터 = 256 KB**(Volta~Hopper 공통). 스레드 하나가
  쓸 수 있는 최대는 **255개**입니다. 레지스터는 점유율(occupancy)을 좌우합니다 — 레지스터 예산이
  상주 워프 수를 제한합니다(스레드당 255개를 다 쓰면 ~8워프로 급감). 단, **상주 워프 상한 자체는
  아키텍처마다 다릅니다** — Volta·A100·Hopper는 SM당 최대 64워프지만 Turing은 32, 컨슈머
  Ampere(GA10x)·Ada는 48워프입니다.
  ([NVIDIA Ada Tuning Guide](https://docs.nvidia.com/cuda/ada-tuning-guide/index.html))
- **공유 메모리 / L1 (온칩 SRAM, SM당)**: Volta 이후 L1 데이터 캐시와 공유 메모리가 **하나의 SRAM
  블록**을 나눠 씁니다(런타임에 분할 비율 선택). A100(GA100)은 합쳐 **192 KB**, 그중 공유 메모리로
  최대 **164 KB**까지; 컨슈머 Ampere(GA10x)·Ada는 **128 KB**(공유 최대 100 KB); Hopper(H100)는
  **256 KB**(공유 최대 228 KB). ([Ampere whitepaper](https://images.nvidia.com/aem-dam/en-zz/Solutions/data-center/nvidia-ampere-architecture-whitepaper.pdf),
  [Hopper Tuning Guide](https://docs.nvidia.com/cuda/hopper-tuning-guide/index.html))
- **L1 / 텍스처 캐시**: SM **전용**. 캐시 라인 = **128 B**, 내부적으로 **32 B 섹터** 4개로 나뉨.
  텍스처 캐시는 [밉맵](./texture-filtering-mipmapping)에서 본 2D 공간 지역성에 최적화돼 있습니다.
- **L2 캐시**: **모든 SM이 공유**하는 단일 캐시. A100 **40 MB**, Ada AD102는 물리 **96 MB**(RTX 4090은
  72 MB 활성), Hopper **50 MB**. L2↔DRAM 트랜잭션 단위는 **32 B 섹터**.
- **VRAM(전역 메모리)**: GB 단위, 가장 크고 가장 느림. 접근 지연은 수백 ns 수준(작업셋·전력상태에
  따라 달라지므로 절대값은 환경 의존).

> **두 독자를 위한 메모.** AMD도 구조는 같습니다 — 이름만 다릅니다. 공유 메모리에 해당하는 것이
> **LDS(Local Data Share)**로, GCN은 CU당 64 KB, RDNA3는 WGP당 128 KB이며 역시 **32뱅크**입니다.
> 워프에 해당하는 실행 묶음은 **wavefront**(GCN 64-lane, RDNA는 32/64). 수치는 다르지만 "작고 빠른
> 온칩 SRAM → 공유 L2 → 큰 VRAM"이라는 계층과, 아래의 합치기·뱅크 충돌 원리는 동일합니다.

이 계층에서 프로그래머가 가장 자주, 가장 크게 손해 보는 두 지점이 **전역 메모리 접근(합치기)**과
**공유 메모리 접근(뱅크 충돌)**입니다. 차례로 봅니다.

## 합치기(coalescing) — 워프의 접근을 트랜잭션으로 묶기

GPU는 스레드 하나씩 메모리를 읽지 않습니다. **워프(32 스레드)가 함께** 메모리에 접근하고,
하드웨어는 그 32개의 주소를 **가능한 한 적은 수의 정렬된 트랜잭션**(32 / 64 / 128 B)으로 묶습니다.
이것이 합치기입니다.

핵심은 **트랜잭션은 통째로 옮겨진다**는 점입니다. 128 B 트랜잭션 하나는, 그 안에서 실제로 쓰는
바이트가 4 B든 128 B든 **항상 128 B를 메모리에서 가져옵니다.** 그러니 워프의 접근이 얼마나 잘 뭉치는지가
유효 대역폭을 직접 결정합니다.

- **이상적(완전 합치):** 32 스레드가 연속된 4 B 워드를 읽으면 주소가 0~127 B에 딱 들어가 **트랜잭션
  1개**로 끝납니다 — 가져온 128 B 전부가 쓸모 있으니 효율 100%.
- **스트라이드 접근:** 스레드마다 멀찍이 떨어진 주소를 읽으면(예: 32 B 간격) 스레드마다 다른
  트랜잭션이 필요해, 32개 트랜잭션 × 128 B = **4096 B를 옮겨 고작 128 B만 쓰는** 꼴이 됩니다.
  효율이 1/스트라이드로 추락합니다.
- **미정렬(misalignment):** 시작 주소가 트랜잭션 경계에 안 맞으면, 같은 데이터를 위해 트랜잭션이
  하나 더 필요해집니다(구형 장치에선 최대 8배까지 대역폭이 깎였습니다).

아래 그림은 워프 32 스레드의 스트라이드·워드 크기·오프셋에 따라, 몇 개의 128 B 트랜잭션으로
묶이는지와 대역폭 효율이 어떻게 달라지는지 보여줍니다.

바꾸는 게 아니라, **버려지는 트랜잭션 바이트를 없애 "유효" 대역폭을 끌어올립니다.** 합치기가 깨진
커널은 같은 일을 하면서 대역폭을 몇 배 더 쓰니, roofline에서 더 깊이 메모리 바운드 영역으로
밀려납니다. "구조는 그대로인데 배열 인덱싱만 바꿨더니 2배 빨라졌다"는 흔한 경험의 정체가 이것입니다.

## 공유 메모리 뱅크 충돌

전역 메모리가 느리니, 자주 쓰는 데이터는 **공유 메모리**(온칩 SRAM)에 올려 재사용합니다. 그런데
공유 메모리에도 함정이 있습니다. 빠른 병렬 접근을 위해 공유 메모리는 **32개의 뱅크(bank)**로
인터리브돼 있습니다. 연속된 32-bit 워드가 연속된 뱅크에 배치됩니다 — 즉 워드의 뱅크 번호는

$$
\text{bank} = \Big(\frac{\text{byte address}}{4}\Big) \bmod 32
$$

각 뱅크는 한 사이클에 **32-bit 하나**를 내줍니다. 그래서:

- 워프의 32 스레드가 **서로 다른 32 뱅크**를 치면 한 번에 다 처리됩니다(충돌 없음).
- 여러 스레드가 **같은 뱅크의 다른 워드**를 치면, 그 접근들은 직렬화됩니다 — $k$개가 한 뱅크에
  몰리면 **$k$-way 충돌**로 $k$배 느려집니다.
- 예외: 여러 스레드가 **정확히 같은 워드**를 읽으면 하드웨어가 한 번 읽어 모두에게 **방송
  (broadcast)**하므로 충돌이 아닙니다.

고전적 함정이 행렬 전치입니다. `32×32` 타일을 공유 메모리에 두고 **열**을 읽으면, 한 열의 32개
원소가 모두 같은 뱅크에 떨어져 **32-way 충돌**이 납니다. 해법은 우아합니다 — 타일 폭을 33으로
**패딩**(`tile[32][33]`)하면 각 행의 뱅크가 한 칸씩 밀려, 열이 32개 뱅크에 고루 퍼집니다. 이
한 줄짜리 트릭으로 전치 커널이 단순 복사 커널의 ~95% 성능까지 회복합니다.
([NVIDIA Matrix Transpose](https://developer.nvidia.com/blog/efficient-matrix-transpose-cuda-cc/))

## 지연 숨기기 — 계층이 점유율과 만나는 곳

전역 메모리 지연은 수백 사이클입니다. GPU는 이 지연을 캐시만으로 숨기지 않습니다 — **상주한 다른
워프로 갈아끼워** 숨깁니다([워프 divergence와 occupancy](./warp-divergence-occupancy) 참고). 한 워프가
메모리를 기다리는 동안 스케줄러는 준비된 다른 워프를 실행합니다. 그래서 이 챕터의 두 자원이
중요합니다: 레지스터·공유 메모리 예산이 **얼마나 많은 워프를 상주시킬 수 있는지**(점유율)를
정하고, 합치기는 **각 워프가 기다려야 할 트랜잭션 수**를 줄입니다. 둘이 함께 메모리 지연을 처리량으로
바꿉니다.

## 정리

- GPU 메모리는 **레지스터 → 공유/L1 → L2 → VRAM**의 계층이고, 위로 갈수록 빠르고 작습니다.
  온칩 자원(레지스터·공유 메모리)이 점유율을 정합니다.
- **합치기**: 워프의 전역 접근은 정렬된 32/64/128 B 트랜잭션으로 묶입니다. 연속·정렬이면 1개로
  끝나 효율 100%, 스트라이드·미정렬이면 트랜잭션이 폭증해 유효 대역폭이 추락합니다.
- **뱅크 충돌**: 공유 메모리는 32뱅크. 같은 뱅크의 다른 워드를 동시에 치면 직렬화됩니다. 패딩으로
  뱅크를 분산시켜 피합니다.
- 이 모든 것이 [Roofline](./memory-bandwidth-roofline)의 "유효 대역폭"을 좌우합니다 — 알고리즘을
  안 바꿔도 메모리 접근 패턴만으로 성능이 수 배 갈립니다.
