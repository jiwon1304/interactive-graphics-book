# DirectX 드라이버 내부 — DX9·DX11·DX12 Under the Hood

세 개의 D3D 버전은 같은 하드웨어, 같은 WDDM 커널 위에서 돕니다. 그런데 동일한 한 장면을
그릴 때 CPU가 하는 일의 양과 위치는 세대마다 크게 다릅니다. DX11에서 draw call 하나가
드라이버 안에서 소비하는 CPU는 흔히 수백 ns~수 µs 수준이고, 프레임당 draw가 1~2만 개를
넘어가면 렌더 스레드 한 코어가 *드라이버 코드만으로* 포화됩니다. GPU는 놀고 있는데 CPU가
병목이 되는 이 상황이 Mantle·DX12·Vulkan을 낳은 직접적 동기입니다.

이 차이는 API 표면이 아니라 **드라이버 뒤에서 무엇을, 어느 레이어가, 언제 하느냐**에서
나옵니다. 그 분담을 레이어부터 따라가 보겠습니다.

## 1. WDDM 스택 — runtime, UMD, KMD

Windows의 그래픽스 드라이버는 Vista의 WDDM(Windows Display Driver Model) 이후 **둘로
쪼개져** 있습니다. 하나는 프로세스 주소공간에서 도는 **UMD**(user-mode display driver,
IHV가 제공하는 DLL — 예: `nvwgf2um.dll`, `amdxc64.dll`), 다른 하나는 커널의
**KMD**(kernel-mode driver, 예: `nvlddmkm.sys`)입니다. 그 사이를 OS의 그래픽스 커널
**Dxgkrnl**이 중재합니다.

<WddmStack client:visible />

호출 흐름은 이렇습니다. 애플리케이션이 `d3d11.dll`/`d3d12.dll` 같은 **D3D runtime**을
호출하면, runtime은 인자·상태를 검증한 뒤 **DDI**(device driver interface)를 통해 UMD를
호출합니다. UMD는 바인딩된 상태와 리소스를 그 하드웨어가 이해하는 **command buffer**(GPU
명령 스트림)로 변환해 프로세스 메모리에 쌓습니다. 셰이더 바이트코드를 GPU ISA로 JIT
컴파일하는 것도 UMD의 일입니다. **여기까지는 전부 user 모드** — 커널 진입이 없습니다.

command buffer를 GPU에 보낼 때만 커널로 내려갑니다. UMD는 `D3DKMT*` 호출로 Dxgkrnl에
제출하고, **VidMM**(video memory manager)이 참조된 allocation들의 **residency**를 보장한
뒤(필요하면 VRAM으로 page-in), **VidSch**(video scheduler)가 그 buffer를 적절한 GPU 엔진의
ring buffer에 큐잉합니다. 주소 처리 방식은 WDDM 세대에 따라 다릅니다. 초기 WDDM(1.x)에서는
GPU가 segment의 *물리주소*를 참조했고, segment가 공유·과할당되며 리소스가 재배치되면 주소가
바뀌므로 UMD가 command buffer마다 **allocation list**와 **patch location list**를 만들어 제출
전에 주소를 patch해야 했습니다. **WDDM 2.0**(Windows 10)부터는 프로세스별 **GPU 가상주소
(GPUVA)** 공간이 생겨, UMD가 가상주소를 직접 기록하고 patch list를 더 이상 만들지 않습니다 —
VidMM은 per-device residency list 기준으로 residency만 책임집니다. DX12는 WDDM 2.0을 요구하고,
최신 Windows에서는 DX9/11 앱도 이 GPUVA 경로로 돕니다.

이 6단 구조는 DX9·11·12가 **공유**합니다. 세 API의 진짜 차이는 이 레이어들 사이에서
*책임을 어떻게 나누느냐*입니다.

## 2. 한 번의 Draw가 GPU에 닿기까지

draw call 하나의 경로를 풀어 봅시다. 핵심은 **per-draw로 반복되는 부분**과 **제출 단위로
분할 상환되는 부분**을 구분하는 것입니다.

<DrawCallPath client:visible />

매 `Draw()`마다 user 모드에서: runtime이 검증하고, DDI로 UMD를 부르고, UMD가 현재 PSO/상태/
바인딩을 하드웨어 명령으로 변환해 command buffer에 append합니다. 이때 UMD는 이 제출이 참조하는
allocation들을 **residency** 목록으로 추적해, VidMM이 실행 전에 resident를 보장하도록 합니다.
(초기 WDDM에서는 여기에 더해 주소 patch용 **patch location list**도 만들었지만, WDDM 2.0의
GPUVA에서는 그 patch가 사라졌습니다 — §1.)

command buffer가 가득 차거나 `Flush`/`Present`가 호출되면 그제서야 **커널로 제출**됩니다.
user→kernel 전환은 비싸지만, 한 번의 제출이 수백~수천 개의 draw를 담으므로 draw당 커널
비용은 분할 상환됩니다. 그래서 **draw call의 CPU 비용은 거의 전부 윗줄(user 모드 변환)**에
있고, DX12가 공략한 지점이 정확히 이 윗줄입니다.

## 3. 상태 변환은 *언제* 일어나는가

세 API를 가르는 가장 큰 축은 **파이프라인 상태를 하드웨어 형식으로 변환하는 시점**입니다.

<StateTranslationTiming client:visible />

**DX9**는 상태를 잘게 흩어진 호출로 받습니다 — `SetRenderState`, `SetSamplerState`,
`SetTexture`. 이 호출들은 값을 기록만 할 뿐이고, 실제 검증과 하드웨어 명령 변환은 **`Draw`
순간에** 일어납니다(draw-time validation). 즉 draw마다 드라이버가 dirty된 상태 전부를 훑어
하드웨어 레지스터 명령으로 풀어내므로, draw당 CPU 비용이 큽니다.

**DX10**이 도입하고 **DX11**이 이어받은 해법이 **immutable state object**입니다. blend,
rasterizer, depth-stencil, sampler 상태를 미리 묶어 객체로 만들어 두면, 변환 비용의 상당
부분이 *생성 시점*으로 당겨집니다. draw 시에는 그 객체를 바인딩만 하면 되죠. 다만 드라이버는
여전히 바인딩을 검사하고 hazard를 추적하므로(다음 절) draw당 비용이 0은 아닙니다.

**DX12**는 이를 끝까지 밀어 **PSO**(pipeline state object)로 만듭니다. 셰이더들, 입력
레이아웃, blend/raster/depth, 렌더 타깃 포맷까지 — 파이프라인을 정하는 거의 모든 것을 하나의
불변 객체로 묶어 `CreatePipelineState`에서 **하드웨어 ISA와 상태로 완전히 컴파일**합니다.
draw 경로에는 `SetPipelineState`로 그 결과를 꽂는 일만 남아, draw당 변환 비용이 사실상
사라집니다. 대가는 PSO 생성이 무겁다는 것 — 이것이 첫 사용 시 hitching의 원인이고, PSO 캐시
(`ID3D12PipelineLibrary`)와 사전 워밍업이 중요한 이유입니다.

셰이더 IR도 함께 진화했습니다. DX9~11은 HLSL을 `fxc`로 **DXBC**로 컴파일하고, UMD가 이를
드라이버 JIT로 GPU ISA로 바꿉니다(셰이더 생성/첫 사용 시). DX12는 `dxc`가 내는 **DXIL**을
쓰며, UMD JIT는 PSO 생성 시점에 ISA를 만듭니다.

## 4. Hazard tracking과 residency — 드라이버가 대신 하던 일

DX9/DX11에서 드라이버가 *조용히* 처리해 주던 두 가지가 DX12에서 앱으로 넘어왔습니다.

**Hazard tracking과 리소스 상태 전이.** 같은 텍스처를 한 draw에서 render target으로 쓰고
다음 draw에서 셰이더 입력(SRV)으로 읽으려면, 그 사이에 쓰기 완료를 기다리고 캐시를 flush하며
리소스 레이아웃을 바꾸는 **배리어**가 필요합니다. DX11 드라이버는 바인딩 변화를 추적해 이런
전이를 자동으로 삽입하고, SRV/RTV 동시 바인딩 같은 위험을 감지해 한쪽을 자동 unbind까지
합니다. DX12는 이 추적을 없애고 앱이 **`ResourceBarrier`**로 전이(`COMMON`→`RENDER_TARGET`
→`PIXEL_SHADER_RESOURCE` 등), UAV 배리어, aliasing 배리어를 명시하게 합니다. 드라이버가
보수적으로 넣던 과도한 배리어가 사라지는 대신, 틀리면 곧장 깨지는 책임도 앱이 집니다.

**Residency.** 어떤 allocation이 지금 VRAM에 있어야 하는지를 DX9/11에서는 VidMM이 참조
빈도에 따라 자동으로 페이지인/아웃합니다. DX12는 `MakeResident`/`Evict`와 DXGI의 메모리
예산(`QueryVideoMemoryInfo`)으로 앱이 직접 residency를 관리할 수 있습니다 — 대형 씬에서
페이징으로 인한 hitch를 앱이 제어하기 위해서입니다.

또 하나, DX11에서 흔한 `Map(D3D11_MAP_WRITE_DISCARD)`로 상수/동적 버퍼를 갱신하면 드라이버는
내부적으로 **버퍼 renaming(versioning)**을 합니다 — GPU가 아직 이전 버전을 읽는 중이어도
stall 없이 새 메모리 블록을 돌려주는 것이죠. DX12에는 이런 자동 renaming이 없어, 앱이 ring
buffer 형태의 upload heap을 직접 굴리고 fence로 수명을 관리합니다.

## 5. 스레딩과 명령 기록

드라이버 오버헤드를 *여러 코어로 나눌 수 있는가*도 세대마다 다릅니다.

<CommandRecordingThreads client:visible />

**DX9**의 device는 사실상 immediate context 하나라, 렌더 명령 기록·제출이 단일 스레드에
묶입니다. 멀티코어가 있어도 draw 제출을 병렬화할 수 없습니다.

**DX11**은 `ID3D11Device`의 자원 생성은 free-threaded로 풀고, 렌더링은 immediate context
하나가 담당하되 **deferred context**로 다른 스레드에서 command list를 기록할 수 있게 했습니다.
그러나 IHV 드라이버가 네이티브 command list(`D3D11_FEATURE_DATA_THREADING.DriverCommandLists`)를
지원하지 않으면, runtime이 deferred 기록을 immediate에서 다시 재생하는 식으로 **에뮬레이트**해
실질 병렬 이득이 제한적이었습니다.

**DX12**는 각 스레드가 자기 **command allocator**(기록 메모리 backing)와 **command list**에
독립적으로 명령을 기록하고, `ExecuteCommandLists`로 큐에 제출합니다. 큐는 **Direct(그래픽스),
Compute, Copy** 세 종류가 있어 비동기 compute·DMA를 별도 엔진으로 보낼 수 있고, 동기화는
**`ID3D12Fence`**로 명시합니다([command queues](command-queues)에서 다룬 fence/배리어 모델이
그대로 적용됩니다). 자주 반복되는 짧은 명령열은 **bundle**로 미리 기록해 재사용할 수도
있습니다. 드라이버 안의 직렬 지점이 사라져, 기록이 코어 수만큼 확장됩니다.

## 6. draw call 비용은 어디로 가는가

지금까지의 분담이 CPU 시간으로 어떻게 나타나는지 봅시다.

<DrawCallCost client:visible />

DX11에서 draw당 비용의 큰 덩어리는 **상태·hazard 검사**와 **디스크립터 패치**입니다 —
드라이버가 매 draw마다 바인딩 테이블을 검사하고 하드웨어 디스크립터로 변환하기 때문입니다.
DX12는 이를 PSO 생성(프레임 밖)으로 옮기고, descriptor heap과 root signature로 디스크립터를
앱이 미리 배치해 두므로 패치가 사라집니다. 남는 것은 명령 기록과 제출뿐이라 draw당 비용이
한 자릿수 분의 일로 줄고, 그만큼 같은 CPU 예산으로 훨씬 많은 draw를 낼 수 있습니다. DX12가
“빠르다”는 표현이 가리키는 것은 GPU 처리량이 아니라 바로 이 **제출 측 CPU 오버헤드**입니다.

## 7. 정리와 더 나아가기

세 API는 같은 WDDM 스택을 쓰지만, 드라이버의 일을 **draw-time에서 creation-time으로,
드라이버에서 애플리케이션으로** 옮겨 온 역사입니다. DX9는 draw 순간에 거의 모든 변환을 하고,
DX11은 state object와 deferred context로 일부를 당기고 나누되 hazard·residency·디스크립터는
드라이버가 쥐고, DX12는 PSO·배리어·descriptor heap·명시적 큐로 그 자동화들을 걷어내 드라이버를
얇게 만들었습니다. 통제와 책임을 맞바꾼 것입니다.

여기서 이어지는 길들:

- **GPU 스케줄링의 진화.** VidSch의 소프트웨어 스케줄링은 WDDM 2.7의 **GPU hardware
  scheduling**으로 일부 하드웨어/펌웨어에 위임됩니다. 제출 지연과 컨텍스트 전환 비용에 영향을
  줍니다.
- **Vulkan / Mantle와의 수렴.** DX12의 PSO·command buffer·explicit barrier·queue 모델은
  Vulkan과 거의 1:1 대응됩니다. 둘 다 같은 문제(드라이버 per-draw 오버헤드)에 대한 같은 답이고,
  [command queues](command-queues) 챕터의 제출/동기화 모델이 그 공통 골격입니다.
- **실행 모델과의 연결.** 드라이버가 만든 command buffer가 GPU에 닿은 *뒤*의 이야기 —
  워프·락스텝·occupancy —는 [GPU 실행 모델](gpu-execution-model)에서 다룹니다. 이 챕터는 그
  명령이 *어떻게 거기까지 가는가*였습니다.
- **상위 추상.** 엔진들은 이 위에 자체 RHI를 얹어 DX11/DX12/Vulkan을 한 인터페이스로
  감쌉니다. 언리얼의 사례는 [언리얼 GPU 프로파일링 & 크래시 디버깅](ue-gpu-crash-debugging)에서
  볼 수 있습니다.
