# 드로우 콜의 일생 — Draw 한 번이 GPU에 닿기까지

`context->Draw(36, 0)`. 한 줄입니다. 이 함수는 보통 수백 ns 안에 리턴하고, 호출한 쪽에서
보면 “삼각형들을 그려라”라는 명령 하나가 GPU로 떠난 것 같습니다. 그런데 이 한 줄이 리턴할 때
**GPU는 아직 아무것도 하지 않았을 가능성이 큽니다.** 더 정확히는, 이 호출은 GPU에 닿기는커녕
**커널 근처에도 가지 않았을** 수 있습니다.

그런데도 이 한 줄이 너무 많이 모이면 게임이 느려집니다. 프레임당 draw가 1~2만 개를 넘어가면
렌더 스레드 한 코어가 *드라이버 코드만으로* 포화되고, GPU는 놀고 있는데 CPU가 병목이 됩니다.
GPU에 닿지도 않은 호출이 어떻게 CPU를 잡아먹을까요? 답은 이 한 줄이 리턴하기까지 **user 모드
안에서 무슨 일이 벌어지는가**, 그리고 그 결과물이 **언제, 어떻게 커널을 거쳐 GPU로 가는가**에
있습니다. 이 챕터는 Draw 한 번이 거치는 파이프라인 단계를 순서대로 살펴봅니다.

> 이 글은 5부작의 3편입니다. **[1편(CPU·GPU의 대화)](gpu-cpu-conversation)** 의 command
> buffer·비동기 제출과, **[2편(WDDM 그래픽스 스택)](wddm-graphics-stack)** 의
> runtime/UMD/KMD·residency·GPU 가상주소(GPUVA)를 읽었다고 가정하고, 그 위에서 “한 Draw의
> 경로”를 자세히 봅니다. 상태/PSO 변환의 *내용*은 4편, D3D11 immediate vs D3D12/Vulkan의
> *깊은 대비*는 5편으로 넘깁니다.

## 1. 한 줄이 리턴하기까지 — user 모드에서 벌어지는 일

`Draw()`를 부르면 가장 먼저 만나는 것은 **D3D runtime**입니다(`d3d11.dll` / `d3d12.dll`).
runtime은 인자와 현재 바인딩이 유효한지 검증합니다 — 정점 수가 버퍼 범위 안인지, 바인딩된
셰이더·렌더 타깃이 서로 호환되는지 같은 것들이죠. 검증을 통과하면 runtime은 **DDI**(device
driver interface)를 통해 **UMD**(user-mode display driver, IHV가 제공하는 DLL)를 호출합니다.

여기서 핵심 작업이 일어납니다. UMD는 지금 바인딩된 상태(셰이더, 렌더 상태, 리소스)를 **그
하드웨어가 직접 이해하는 명령**으로 변환해, 프로세스 메모리에 있는 **command buffer**에
append합니다. 이 변환이 draw call CPU 비용의 큰 덩어리입니다 — 그리고 **이 전부가 user
모드에서** 일어납니다. 여기까지 커널 진입은 한 번도 없습니다. `Draw()`가 빨리 리턴하는 이유가
이것입니다. 명령을 *적었을* 뿐, GPU에 *보내지* 않았으니까요.

<DrawCallPath client:visible />

도식의 윗줄(검증 → DDI → UMD 변환 → append)이 **매 draw마다** 도는 부분입니다. command
buffer는 이렇게 draw 명령들이 차곡차곡 쌓이는 user 메모리 영역이고, 한 프레임에 수천 개의
draw가 같은 버퍼(또는 버퍼들)에 누적됩니다.

## 2. 언제 커널로 내려가는가 — 제출(submission)

쌓인 command buffer는 어느 순간 GPU로 가야 합니다. 그 “어느 순간”은 매 draw가 아닙니다.
**command buffer가 가득 차거나, 앱이 `Flush`를 부르거나, `Present`로 프레임을 끝낼 때** 비로소
커널로 내려갑니다. 이 내려가는 행위가 **제출(submission)** 이고, 여기서 처음으로 **user→kernel
전환**이 일어납니다.

WDDM 2.0(GPUVA를 쓰는 현대 경로)에서 UMD는 `D3DKMTSubmitCommand`로 graphics kernel
(Dxgkrnl)에 command buffer를 넘깁니다. Microsoft의 드라이버 문서는 이 함수를 이렇게 설명합니다 —
*“이 컨텍스트들은 명령을 user 모드에서 직접 생성하고, 자신의 command buffer 풀을 관리하며,
allocation list나 patch location list를 쓰지 않는다.”* 2편에서 본 GPUVA의 결과가 여기 그대로
나타납니다. UMD가 가상주소를 직접 기록하므로, 옛 WDDM처럼 제출 전에 주소를 patch하는 단계가
없습니다. UMD가 커널에 마저 알려야 할 부가 정보는 별도의 **private driver data buffer**로 함께
보냅니다.

커널로 내려가면 2편에서 본 단계들이 이어집니다. **VidMM**이 이 command buffer가 참조하는
allocation들의 **residency**를 보장하고(필요하면 VRAM으로 page-in), **VidSch**가 그 buffer를
적절한 GPU 엔진의 ring buffer에 큐잉합니다. 그제서야 GPU가 실제로 명령을 꺼내 실행하기
시작합니다. GPUVA에서 UMD는 allocation/patch list 대신 **per-device residency list**만 관리하고,
VidMM이 그 list를 기준으로 실행 전에 resident를 보장합니다.

여기서 비용 구조의 핵심이 드러납니다. user→kernel 전환은 비쌉니다 — 컨텍스트 전환, 인자 복사,
커널 측 검증이 따라붙죠. 그런데 **한 번의 제출이 수백~수천 개의 draw를 담은 command buffer를
한꺼번에** 넘깁니다. 그래서 비싼 커널 전환 비용은 그 안의 draw들로 **분할 상환(amortize)** 됩니다.
이것이 1편에서 본 “비동기·배치” 원리의 비용 측면입니다.

### 제출을 줄이면 draw당 커널 비용이 떨어진다

이 분할 상환을 그림으로 봅시다. 아래 위젯은 한 프레임의 draw 수를 고정하고, **제출당 draw
수**만 달리해 그 효과를 보여줍니다.

<RecordVsSubmit client:visible />

제출당 draw 수가 왼쪽 끝(=1)인 경우는 “draw마다 command buffer를 닫고 커널로 내려간다”는 극단입니다.
한 프레임에 수천 번의 user→kernel 전환이 일어나, 비싼 커널 비용이 프레임 시간을 압도합니다.
오른쪽으로 갈수록 한 제출이 더 많은 draw를 담아 제출 횟수가 줄고, **draw당 커널 비용**이 급격히
떨어집니다. 기록(user 변환) 비용은 draw 수에 정비례하므로 변하지 않습니다 — 줄어드는 것은 오직
*제출 측* 비용입니다.

이 원리는 D3D만의 것이 아닙니다. Vulkan 명세는 제출 함수 `vkQueueSubmit`에 대해 명시적으로
경고합니다 — *“Submission can be a high overhead operation, and applications should attempt to
batch work together into as few calls to `vkQueueSubmit` … as possible.”* 제출은 비싸니
가능한 한 적게, 한 번에 묶어 부르라는 것입니다. 같은 비용 구조에서 나온 같은 처방입니다.

## 3. 그래서 draw call은 *어디서* 비싼가

GPU에 닿지도 않은 `Draw()`가 CPU를 잡아먹는다는 모순은 이제 풀립니다. 비용의 거의 전부는
**제출 전, user 모드 변환**에 있습니다. 그 한 draw의 user CPU 비용을 범주로 쪼개 봅시다.

<DrawCostBreakdown client:visible />

한 draw의 user CPU는 대략 네 곳으로 흩어집니다. **검증**(runtime이 인자·바인딩이 유효한지
본다), **상태·hazard tracking**(드라이버가 매 draw마다 바인딩을 검사하고, 같은 리소스를 쓰며
읽는 것 같은 위험을 추적한다), **디스크립터 패치**(바인딩을 그 하드웨어의 디스크립터 형식으로
변환한다), 그리고 **명령 기록**(변환 결과를 command buffer에 적는다). 제출은 §2에서 봤듯 분할
상환되어 per-draw로는 작으므로 여기선 생략했습니다.

> ⚠️ 위젯의 ns 수치는 특정 드라이버 측정값이 아니라 **구성비·기울기를 보이기 위한 도식용 대표
> 차수**입니다. 절대값이 아니라 “한 draw가 네 곳에 시간을 흘리고, 그게 draw 수에 곱해진다”는
> 관계가 요점입니다.

아래 막대에서 보듯, 이 per-draw 합에 프레임당 draw 수가 곱해지면 금세 60fps의 **16.6ms 예산**을
넘습니다. GPU가 한가해도 CPU가 이 변환·검사·기록을 다 못 끝내면 프레임이 늦어지는 것입니다.
DirectX-Specs의 D3D12 CPU Efficiency 문서가 D3D11의 이 비용을 “order of magnitude”로 줄이는
것을 목표로 삼은 이유가 여기 있습니다. 그 “어떻게”(상태를 PSO로 미리 컴파일, 디스크립터를 앱이
미리 배치) 는 4·5편의 주제입니다. 이 챕터는 *어디가 비싼지*까지입니다.

## 4. 기록이 곧 변환인가 — D3D11 immediate vs D3D12/Vulkan

§1에서 “UMD가 draw를 하드웨어 명령으로 *변환*한다”고 했는데, 그 변환이 **언제** 일어나는가는
API마다 다릅니다. 이 차이가 5편의 큰 주제지만, 기록 경로의 모양만 먼저 나란히 봐 둡시다.

<D3dVsVulkanRecord client:visible />

**D3D11 immediate context**에서는 `Draw()`를 부르는 *그 순간* 드라이버가 현재 상태를 하드웨어
명령으로 변환합니다(draw-time 변환). 변환이 매 draw에 묶여 있고, 이 immediate context는 사실상
하나라 변환을 여러 코어로 나누기 어렵습니다.

**D3D12 command list와 Vulkan VkCommandBuffer**는 다릅니다. Vulkan에서 command buffer는
`vkBeginCommandBuffer`로 기록(recording) 상태에 들어가고, `vkCmdDraw` 같은 호출은 명세가
“Action Command”라고 부르는 **기록 명령**입니다 — 드라이버에게 즉시 변환을 시키는 게 아니라,
명령을 **command buffer에 직접 적습니다.** 다 적으면 `vkEndCommandBuffer`로 닫고, 나중에
`vkQueueSubmit`(D3D12는 `ExecuteCommandLists`)으로 **배치 제출**합니다. 그래서 여러 스레드가
각자의 command buffer에 동시에 기록할 수 있습니다.

이 구조에서 “D3D11이 draw마다 하던 드라이버 변환”의 상당 부분은 두 곳으로 흩어집니다 — 일부는
파이프라인 상태를 미리 묶는 객체(PSO)의 *생성 시점*으로, 일부는 *기록 시점*의 가벼운 append로요.
그래서 DirectX-Specs는 D3D12에서 *“execution-time processing is generally less than 10% of
overall command list CPU usage”* 라고 말합니다 — CPU 비용의 대부분이 *제출*이 아니라 *기록*에
있다는 뜻이고, 그 기록을 코어 수만큼 병렬화할 수 있다는 게 핵심 이득입니다. 변환의 *내용*이
어떻게 PSO 생성으로 당겨지는지(4편)와, immediate vs deferred 기록의 *전체 대비*(5편)는 다음
글들에서 이어집니다.

## 5. 정리 — 한 Draw가 거치는 단계

`Draw()` 한 번이 거치는 단계를 한 줄로 요약하면 이렇습니다. **호출 → runtime 검증 → DDI로 UMD 호출 →
UMD가 상태를 하드웨어 명령으로 변환해 command buffer에 append**(여기까지 전부 user 모드, 매
draw 반복) **→ buffer full 또는 Flush/Present 시 `D3DKMTSubmitCommand`로 커널 제출**(user→kernel
전환, 제출 단위) **→ VidMM이 residency 보장 → VidSch가 ring buffer에 큐잉 → GPU 실행.**

비용 구조의 두 축을 기억하면 충분합니다.

- **per-draw(user CPU)**: 검증·상태/hazard·디스크립터·기록. draw 수에 정비례하고, draw call이
  “비싸다”는 말이 가리키는 거의 전부입니다.
- **제출 단위(kernel)**: user→kernel 전환과 residency·스케줄. 비싸지만 제출당 draw 수로 분할
  상환되므로, 제출을 적게 묶을수록 draw당 비용이 떨어집니다.

여기서 이어지는 길들:

- **상태 변환의 *내용* (4편).** UMD가 “상태를 하드웨어 명령으로 변환”한다고 했지만, 그 변환이
  구체적으로 무엇을 하는지 — 셰이더 ISA, 레지스터 패킹, 디스크립터 레이아웃 — 와 그것이 PSO
  생성으로 어떻게 당겨지는지를 다룹니다.
- **immediate vs explicit의 전체 대비 (5편).** D3D11이 draw마다 변환하던 모델과 D3D12/Vulkan의
  미리-기록 모델을, 스레딩·드라이버 두께·CPU 오버헤드 관점에서 끝까지 비교합니다. D3D12 Runtime
  Bypass처럼 runtime 자체까지 얇게 만드는 시도도 여기에 들어갑니다.
- **이미 다룬 토대.** 제출 후 커널 안에서 벌어지는 일(residency·스케줄)은
  [2편(WDDM 그래픽스 스택)](wddm-graphics-stack), command buffer·비동기 제출의 큰 그림은
  [1편(CPU·GPU의 대화)](gpu-cpu-conversation)에 있습니다.
